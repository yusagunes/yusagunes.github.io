dbt'de Incremental Modeller — Doğru Stratejiyi Seçmek

merge, insert_overwrite, append. Her durum için hangi strateji daha iyi çalışır?

dbt Best Practices
🔧

dbt ile çalışırken en sık düştüğümüz tuzaklardan biri: incremental modeli doğru strateji olmadan kurmak. Tablo her gün milyonlarca satır büyüyorsa ve sen full refresh yapıyorsan, hem warehouse'e hem cebe zarar veriyorsun. Bu yazıda üç ana stratejiyi ve ne zaman hangisini seçmen gerektiğini konuşuyoruz.

1. append — Sadece Ekle

En basit strateji: yeni satırları tablonun sonuna ekle. Geçmiş veri hiç değişmez. 

{{ config(materialized='incremental', incremental_strategy='append') }}

select *
from {{ source('events', 'raw_events') }}
{% if is_incremental() %}
  where event_date > (select max(event_date) from {{ this }})
{% endif %}

Ne zaman: Event log'ları, append-only audit tabloları, immutable veri kaynakları.

Ne zaman değil: Güncellenebilen veriler (status değişen siparişler gibi).

2. merge — Upsert

Satırları unique key ile eşleştirir, varsa günceller yoksa ekler. En esnek ama en maliyetli. 

{{ config(
    materialized='incremental',
    incremental_strategy='merge',
    unique_key='order_id'
) }}

select *
from {{ ref('stg_orders') }}
{% if is_incremental() %}
  where updated_at > (select max(updated_at) from {{ this }})
{% endif %}

Ne zaman: SCD Type 1, güncellenebilen kayıtlar (orders, users, inventory).

Trade-off: Her merge çalıştırmasında warehouse'de delete+insert olur. Çok büyük tablolarda yavaş olabilir.

3. insert_overwrite — Partition Bazlı

BigQuery, Redshift, Spark gibi partition'lı warehouse'lerde çok güçlü. Sadece belirli partition'ları siler ve yeniden yazar. 

{{ config(
    materialized='incremental',
    incremental_strategy='insert_overwrite',
    partition_by={'field': 'event_date', 'data_type': 'date'},
    partitions=['event_date >= "2026-06-01"']
) }}

select *
from {{ source('events', 'raw_events') }}
where event_date >= '2026-06-01'

Ne zaman: Tarih bazlı event tabloları, partitioning'in doğal yapıldığı durumlar.

Avantaj: Merge'den çok daha ucuz (sadece ilgili partition'lar yeniden yazılır).

Hangisini seçeyim?

append — Veri hiç güncellenmiyorsa

merge — Satırlar güncelleniyor, history önemli değilse

insert_overwrite — Partition bazlı warehouse + tarih bazlı güncelleme

Sık yapılan hatalar

  • is_incremental() unutmak: İlk çalıştırmada tüm veri, sonrakilerde incremental — bunu kontrol eden is_incremental() makro'sudur.
  • Yanlış unique_key: Merge'de unique_key tek olmalı. Composite key için ['order_id', 'line_item_id'] gibi array verebilirsin.
  • Lookup sorgusunu yanlış yazmak: select max(updated_at) from this çok pahalı olabilir. Bir "watermark" tablosu tutabilirsin.

Sonuç

Doğru incremental strateji seçmek, hem performans hem de maliyet açısından kritik. Küçük tablolarda fark etmez, ama milyonlarca satıra ulaştığında ciddi fark yaratır. Verinin doğasını iyi anlamak, doğru stratejiyi seçmenin yarısı.
