PySpark DataFrame Optimizasyonu — 5 Altın Kural

Partition stratejisi, broadcast join, predicate pushdown. PySpark işlerini 10x hızlandıran pratikler.

Spark Performance Python
⚡

PySpark ile büyük veri işleri yazarken, kodun "çalışıyor olması" yetmez. Onlarca GB'lık veri üzerinde koşan bir job'un 30 dakika mı yoksa 3 dakika mı sürdüğü, tamamen yazdığın küçük detaylara bağlı. Bu yazıda, production'da job sürelerini ciddi şekilde düşüren 5 pratiği paylaşıyorum.

1. Predicate Pushdown'u Doğru Kullan

Spark, filtreleri mümkün olduğunca erken uygulamaya çalışır. Ama bu her zaman olmaz. Özellikle Parquet gibi columnar formatlarda, okuma anında filtreyi aşağı itmek büyük fark yaratır. 

df = spark.read.parquet("data/events/")
df.filter(F.col("event_date") >= "2026-01-01") \
  .filter(F.col("country") == "TR") \
  .groupBy("user_id") \
  .agg(F.count("*").alias("event_count"))

Bu sorgu, tüm dosyaları okumadan önce partition pruning ve predicate pushdown tetikler. Sorgu planını df.explain(True) ile kontrol et — PushedFilters görüyorsan doğru yoldasın.

2. Broadcast Join: Küçük Tablolar İçin Hayat Kurtarır

Default olarak Spark, join'lerde sort-merge yapar. Büyük + küçük tabloyu birleştiriyorsan, küçük tabloyu broadcast etmek shuffle'ı tamamen ortadan kaldırır. 

from pyspark.sql.functions import broadcast

result = large_df.join(
    broadcast(small_df),  # ~200MB'tan küçük tablolar için ideal
    on="user_id",
    how="left"
)

Kural: spark.sql.autoBroadcastJoinThreshold default 10MB. Bunu ihtiyacına göre ayarla, ama 200MB üstü için broadcast yapma — executor memory yetersiz kalır.

Pratik ipucu: Broadcast threshold'unu Spark UI'da shuffle okumalarını görerek ayarla. Çok az shuffle = çok hızlı job.

3. Partition Sayısını Doğru Tut

Çok fazla partition → task scheduling overhead. Çok az → executor'lar boş bekler. İyi bir kural: her partition ~128MB-256MB veri içersin. 

# Yeniden partition
df = df.repartition(200, "user_id")

# Veya coalesce ile birleştir
df = df.coalesce(50)

repartition shuffle yapar (ama paralelliği artırır), coalesce shuffle'sız birleştirir (sadece küçültmek için).

4. Cache'i Akıllı Kullan

Bir DataFrame'i birden fazla action'da kullanıyorsan cache'le. Ama her şeyi cache'leme — memory şişer, GC patlar. 

from pyspark.storagelevel import StorageLevel

df_filtered = df.filter(...).cache()  # Sadece tekrar kullanacaksan
df_filtered.count()  # materialization tetikler
result1 = df_filtered.groupBy(...).agg(...)
result2 = df_filtered.join(...)

df_filtered.unpersist()  # iş bitince bırak

5. UDF Yerine Pandas/Python Built-in Kullan

Python UDF'ler JVM ↔ Python arasında veri taşır. Bu yavaştır. Mümkün olduğunca Spark SQL fonksiyonlarını veya pandas UDF kullan. 

# YAVAŞ — Python UDF
@udf("string")
def normalize(s):
    return s.strip().lower() if s else None

# HIZLI — Native SQL function
from pyspark.sql.functions import lower, trim
df.withColumn("name_norm", lower(trim(F.col("name"))))

Sonuç

Bu 5 kural, kendi production job'larımda 8x ile 15x arası hızlanma sağladı. Hangisinin en büyük etkiyi yarattığı veriye göre değişir ama predicate pushdown ve broadcast join hemen her iş yükünde fark yaratır.

Bir sonraki yazıda: Spark UI okuma — executor memory, shuffle spill ve skewed partition tespiti üzerine.
